# openppp2 编译环境与依赖清单（基于原版代码）

> 本文档基于原版 [liulilittle/openppp2](https://github.com/liulilittle/openppp2) 代码，以 CI 工作流为准记录编译环境与构建步骤。
> 构建目标：WSS 修改版 Debug（本项目 Debug），保留调试日志。
> 最后更新：2026-06-24
>
> **相比原版的修改内容（详见附录）：**
> - 完全静态链接（`-static -no-pie -nodefaultlibs`）
> - GLIBC 兼容层（`glibc_compat.h` + `libglibc_compat.a`）
> - 条件编译（`ENABLE_IO_URING` / `ENABLE_TC` / `NOT_HAVE_SIMD`）
> - 多变体构建（amd64 × 7 + aarch64 × 4）
> - CI/CD 工作流（GitHub Actions 矩阵构建）
> - 新增 `client.websocket.host` / `client.websocket.sni` 优选 IP 支持

## 三、GLIBC 兼容层

### 3.1 背景

OpenSSL 3.0.15 在 GLIBC 2.38+ 环境下编译时，以下 6 个源文件定义了 `_GNU_SOURCE` 宏，导致引用了 GLIBC_2.38 新增的 `__isoc23_*` 系列函数：

- `crypto/bio/bss_dgram.c`
- `crypto/bio/bio_addr.c`
- `crypto/getenv.c`
- `crypto/dso/dso_dlfcn.c`
- `engines/e_afalg.c`
- `providers/implementations/rands/seeding/rand_unix.c`

### 3.2 解决方案

提供兼容层实现，将 `__isoc23_*` 函数委托给标准 C 函数。

#### 源码文件：`glibc_compat.h`

位置：项目根目录 `glibc_compat.h`

```c
// GLIBC compatibility shim: provide __isoc23_* symbols that delegate to
// the original GLIBC functions. This allows the binary to run on systems
// with GLIBC < 2.38 even when OpenSSL static library references __isoc23_*.
#if defined(__linux__) && defined(__GLIBC__)

#include <stdlib.h>
#include <stdarg.h>

long __isoc23_strtol(const char* __restrict nptr, char** __restrict endptr, int base) {
    return strtol(nptr, endptr, base);
}

long long __isoc23_strtoll(const char* __restrict nptr, char** __restrict endptr, int base) {
    return strtoll(nptr, endptr, base);
}

unsigned long __isoc23_strtoul(const char* __restrict nptr, char** __restrict endptr, int base) {
    return strtoul(nptr, endptr, base);
}

unsigned long long __isoc23_strtoull(const char* __restrict nptr, char** __restrict endptr, int base) {
    return strtoull(nptr, endptr, base);
}

int __isoc23_sscanf(const char* __restrict s, const char* __restrict format, ...) {
    int result;
    va_list args;
    va_start(args, format);
    result = vsscanf(s, format, args);
    va_end(args);
    return result;
}

#endif
```

#### 使用方式

在 `main.cpp` 顶部添加：
```cpp
#include <glibc_compat.h>
```

#### 独立静态库版本（备用）

如果不想修改 `main.cpp`，可以编译为独立静态库：

```bash
# 源码：/tmp/glibc_compat_standalone.c
gcc -c -o /tmp/glibc_compat.o /tmp/glibc_compat_standalone.c
ar rcs /tmp/libglibc_compat.a /tmp/glibc_compat.o
```

然后在 CMakeLists.txt 的 `TARGET_LINK_LIBRARIES` 中添加 `/tmp/libglibc_compat.a`。

#### 链接器标志（Linux 完全静态）

CMakeLists.txt 中通过条件判断设置链接器标志，确保无论是否通过 toolchain 文件预设了 `CMAKE_EXE_LINKER_FLAGS`，都能正确追加静态链接参数：

```cmake
# Linux: 完全静态链接（含条件判断，兼容 toolchain 预设）
IF(NOT CMAKE_EXE_LINKER_FLAGS)
    SET(CMAKE_EXE_LINKER_FLAGS "-static -no-pie -Wl,--gc-sections -s -nodefaultlibs -rdynamic -Wl,-Bstatic")
ELSE()
    STRING(REPLACE "-rdynamic" "" CMAKE_EXE_LINKER_FLAGS "${CMAKE_EXE_LINKER_FLAGS}")
    SET(CMAKE_EXE_LINKER_FLAGS "${CMAKE_EXE_LINKER_FLAGS} -rdynamic -Wl,-Bstatic")
ENDIF()

# macOS: 动态链接
# SET(BUILD_SHARED_LIBS ON)

# Clang: 动态链接
# SET(BUILD_SHARED_LIBS ON)
```

> **关键修复**：`-rdynamic -Wl,-Bstatic` 必须追加在已有标志之后，否则 aarch64 toolchain 预设的 `-L` 路径会被覆盖，导致链接失败。

### 4.3 重要注意事项

1. **`main.cpp` 不能通过 `FILE(GLOB_RECURSE ...)` 匹配**：否则会递归匹配到 `vcpkg/buildtrees/` 目录下的同名文件，导致编译错误。必须在 `ADD_EXECUTABLE` 中直接指定。

2. **`-nodefaultlibs` 必须使用**：GCC 的 specs 文件在 `-static` 模式下仍会引用 `.so` 文件路径。使用 `-nodefaultlibs` 阻止 GCC 自动添加默认库，然后手动通过 `-Wl,-Bstatic` 指定所有静态库。

3. **`libm.a` 空桩问题**：Deepin 23 的 `libm.a` 是 132 字节的空桩，需要替换为 `libc.a` 的软链接：
   ```bash
   ln -sf /usr/lib/x86_64-linux-gnu/libc.a /usr/lib/x86_64-linux-gnu/libm.a
   ```

4. **Boost 1.87.0 API 变更**：
   - `io_service` → Boost 1.87.0 已移除，代码统一使用 `io_context`。`stdafx.h` 中通过版本判断做后向兼容：`BOOST_VERSION < 108700` 时定义 `typedef io_service io_context`（即旧版 Boost 将 `io_service` 别名到 `io_context`，新版 Boost 原生使用 `io_context`）
   - `basic_resolver::iterator` → 使用 `results_type`
   - `io_context::work` → 使用 `executor_work_guard`
   - `io_context::run(ec)` → 使用 `run()` 无参数版本

5. **`CMAKE_EXE_LINKER_FLAGS` 条件判断**：CMakeLists.txt 使用 `IF(NOT CMAKE_EXE_LINKER_FLAGS)` 判断是否已预设链接器标志。当使用 aarch64 toolchain 文件时，toolchain 中设置的 `CMAKE_EXE_LINKER_FLAGS` 会触发 `ELSE()` 分支，追加 `-rdynamic -Wl,-Bstatic` 而非覆盖。因此 toolchain 文件**必须**包含完整的 `-static -no-pie` 等标志。

6. **aarch64 交叉编译时 `-DTHIRD_PARTY_LIBRARY_DIR` 必须指定**：aarch64 的第三方库位于 `/root/dev/arm64/`，与 amd64 的 `/root/dev/` 不同。编译 aarch64 时必须传入 `-DTHIRD_PARTY_LIBRARY_DIR=/root/dev/arm64`。

7. **`glibc_compat.h` 已包含在 `main.cpp` 中**：项目根目录的 `glibc_compat.h` 已在 `main.cpp` 顶部通过 `#include <glibc_compat.h>` 包含，无需额外链接 `/tmp/libglibc_compat.a`。CMakeLists.txt 中同时保留了 `${CMAKE_SOURCE_DIR}/libglibc_compat.a` 作为备用链接方式。

### 5.3 amd64 变体编译

项目支持 8 种 amd64 变体，通过 CMake 选项组合控制：

### 5.4 交叉编译（aarch64）

aarch64 编译需要使用 toolchain 文件，并指定 aarch64 版本的第三方库路径。

#### aarch64 Toolchain 文件

创建 `/tmp/arm64-toolchain.cmake`：

```cmake
set(CMAKE_SYSTEM_NAME Linux)
set(CMAKE_SYSTEM_PROCESSOR aarch64)

set(CMAKE_C_COMPILER aarch64-linux-gnu-gcc)
set(CMAKE_CXX_COMPILER aarch64-linux-gnu-g++)

set(CMAKE_FIND_ROOT_PATH /root/dev/arm64 /usr/lib/aarch64-linux-gnu /usr/include)
set(CMAKE_FIND_ROOT_PATH_MODE_PROGRAM NEVER)
set(CMAKE_FIND_ROOT_PATH_MODE_LIBRARY ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_INCLUDE ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_PACKAGE ONLY)

# 必须包含 -static -no-pie 等标志，因为 CMakeLists.txt 中的 IF(NOT CMAKE_EXE_LINKER_FLAGS)
# 判断会因 toolchain 预设了该变量而进入 ELSE() 分支，仅追加 -rdynamic -Wl,-Bstatic
set(CMAKE_EXE_LINKER_FLAGS_INIT "-static -no-pie -Wl,--gc-sections -s -rdynamic -Wl,-Bstatic -L/usr/lib/aarch64-linux-gnu -L/usr/lib/gcc-cross/aarch64-linux-gnu/12")
set(CMAKE_TRY_COMPILE_TARGET_TYPE STATIC_LIBRARY)
```

> **注意**：
> - 使用 `CMAKE_EXE_LINKER_FLAGS_INIT` 而非 `CMAKE_EXE_LINKER_FLAGS`，避免 CMake 内部测试编译因 `-nodefaultlibs` 失败。
> - `CMAKE_TRY_COMPILE_TARGET_TYPE STATIC_LIBRARY` 跳过编译测试，避免 toolchain 标志导致测试失败。
> - 不能使用 `-nodefaultlibs`，因为 aarch64 交叉编译器的 specs 处理方式不同。

#### aarch64 基础版编译

```bash
cd /root/build/openppp2

cmake .. \
    -DCMAKE_TOOLCHAIN_FILE=/tmp/arm64-toolchain.cmake \
    -DNOT_HAVE_SIMD=ON \
    -DTHIRD_PARTY_LIBRARY_DIR=/root/dev/arm64
make -j$(nproc)
```

## 八、常见问题

### 8.1 链接错误：`attempted static link of dynamic object`

**原因**：GCC 在 `-static` 模式下仍引用 `.so` 文件。

**解决**：使用 `-nodefaultlibs` 阻止 GCC 自动添加默认库，然后手动通过 `-Wl,-Bstatic` 指定所有静态库。

### 8.2 链接错误：`undefined reference to __isoc23_*`

**原因**：OpenSSL 3.0.15 在 GLIBC 2.38+ 环境下编译引用了 GLIBC_2.38 符号。

**解决**：在 `main.cpp` 中包含 `glibc_compat.h`，或在链接时添加 `/tmp/libglibc_compat.a`。

### 8.3 运行时错误：`GLIBC_2.34 not found`

**原因**：二进制在旧 GLIBC 系统上运行，但引用了新 GLIBC 符号。

**解决**：使用完全静态链接（本文档的配置），消除所有 GLIBC 动态依赖。

### 8.4 编译错误：`vcpkg/buildtrees/` 下的 `main.cpp` 被编译

**原因**：`FILE(GLOB_RECURSE ...)` 递归匹配了 `vcpkg/` 目录。

**解决**：将 `main.cpp` 从 `GLOB_RECURSE` 中移除，直接在 `ADD_EXECUTABLE` 中指定。

### 8.5 Boost 编译错误：`io_service` 未定义

**原因**：Boost 1.87.0 移除了 `io_service`。

### 8.6 链接产物是 PIE（DYN 类型）而非 EXEC

**原因**：`CMAKE_EXE_LINKER_FLAGS` 中缺少 `-no-pie`，或 toolchain 文件覆盖了该标志。

**解决**：
- amd64：确保 CMakeLists.txt 的 `IF(NOT CMAKE_EXE_LINKER_FLAGS)` 分支包含 `-no-pie`
- aarch64：确保 toolchain 文件的 `CMAKE_EXE_LINKER_FLAGS_INIT` 包含 `-no-pie`

### 8.7 aarch64 编译报错：`skipping incompatible` 库文件

**原因**：`THIRD_PARTY_LIBRARY_DIR` 指向了 amd64 的 `/root/dev`，而非 aarch64 的 `/root/dev/arm64`。

**解决**：编译 aarch64 时必须传入 `-DTHIRD_PARTY_LIBRARY_DIR=/root/dev/arm64`。

### 8.8 aarch64 编译报错：`cannot find -lssl` / `cannot find -lcrypto`

**原因**：OpenSSL aarch64 库路径未被正确搜索到。

**解决**：检查 toolchain 文件的 `CMAKE_FIND_ROOT_PATH` 是否包含 `/root/dev/arm64`，以及 `LINK_DIRECTORIES` 是否正确指向 `${THIRD_PARTY_LIBRARY_DIR}/openssl`。

### 8.9 aarch64 CMake 配置报错：`The C compiler is not able to compile a simple test program`

**原因**：toolchain 文件中使用了 `-nodefaultlibs` 导致编译测试程序失败。

**解决**：使用 `CMAKE_EXE_LINKER_FLAGS_INIT` 替代 `CMAKE_EXE_LINKER_FLAGS`，并添加 `set(CMAKE_TRY_COMPILE_TARGET_TYPE STATIC_LIBRARY)` 跳过编译测试。

### 8.10 编译产物在旧 GLIBC 系统上运行报错：`undefined symbol: malloc, version GLIBC_2.2.5`

**原因**：产物是 PIE 动态链接（DYN 类型），运行时动态链接器尝试解析 GLIBC 符号。

**解决**：确保产物是 `EXEC (Executable file)` 类型且 `statically linked`。检查 `CMAKE_EXE_LINKER_FLAGS` 是否包含 `-static -no-pie`。



**解决**：在 `stdafx.h` 中添加版本判断 typedef：
```cpp
#include <boost/version.hpp>
#if BOOST_VERSION >= 108700
namespace boost { namespace asio { using io_service = io_context; } }
#endif
```

## 九、WebSocket/WebSocket Secure (WS/WSS) 功能修改记录

> 本章记录 2026-06-22 ~ 2026-06-23 期间为支持 WS/WSS 优选 IP 连接所做的全部修改。

### 9.1 功能概述

#### 9.1.1 背景

原始实现中，WebSocket 连接的 Host 和 TLS SNI 固定使用 URL 中的主机名。当使用优选 IP 连接时（例如用 `23.249.18.223` 连接，但需要发送 `nube-hk.4040404.top` 作为 Host 和 SNI），无法自定义这两个字段，导致 TLS 握手失败或 WebSocket 被服务端拒绝。

#### 9.1.2 解决方案

在客户端配置中新增 `websocket` 子配置项，允许独立指定 WebSocket Host 和 TLS SNI：

```json
{
    "client": {
        "server": "wss://23.249.18.223:46800/tun",
        "websocket": {
            "host": "nube-hk.4040404.top",
            "sni": "nube-hk.4040404.top"
        }
    }
}
```

- **`host`**：WebSocket HTTP Host 头字段。为空时使用 URL 中的主机名或服务端配置的 `websocket.host`。
- **`sni`**：TLS SNI (Server Name Indication) 扩展字段。为空时使用 `host` 的值。

#### 9.1.3 数据流

```
客户端配置 (appsettings.json)
    │
    ├── client.websocket.host ──→ WebSocket HTTP Host 头
    │
    └── client.websocket.sni  ──→ TLS SNI (SSL_set_tlsext_host_name)
                                      │
                                      ↓
                               SslSocket::Run()
                               SSL_set_tlsext_host_name(sni)
```

### 9.2 服务端 TCP 转发（VirtualEthernetExchanger）

#### 9.2.1 背景

在 `mux=true` 模式下（WSS 隧道使用此模式），服务端使用 `VirtualEthernetExchanger` 处理客户端请求。原始实现中，`OnConnect()`、`OnPush()`、`OnDisconnect()` 全部返回 `false`，导致：

1. 客户端发起的 TCP 连接请求被忽略
2. `PacketInput()` 返回 `false` → `Run()` 循环退出 → Exchanger 被销毁
3. WSS 隧道实际上无法转发 TCP 数据

#### 9.2.2 解决方案

在 `VirtualEthernetExchanger` 中实现了完整的 TCP 转发功能：

**新增数据结构：**
```cpp
struct TcpConnection {
    std::shared_ptr<boost::asio::ip::tcp::socket>  Socket;     // 出站 TCP 套接字
    std::shared_ptr<Byte>                           Buffer;     // 接收缓冲区
    bool                                            Connected = false;
};
typedef ppp::unordered_map<int, std::shared_ptr<TcpConnection>> TcpConnectionTable;
```

**新增方法：**
| 方法 | 功能 |
|------|------|
| `OnConnect(transmission, connection_id)` | 创建出站 TCP 连接，连接到目标地址，发送 CONNECT_OK |
| `OnPush(transmission, connection_id, data)` | 将客户端数据转发到出站 TCP 套接字 |
| `OnDisconnect(transmission, connection_id)` | 关闭并清理 TCP 连接 |
| `DeleteTcpConnection(connection_id)` | 从映射表中移除并关闭连接 |
| `ReceiveTcpConnection(conn, connection_id)` | 从出站套接字读取数据并转发回传输层 |

**协议格式：**
```
[action(1)] [connection_id(3)] [payload...]

action:
  0x2A = SYN      (连接请求)
  0x2B = SYNOK    (连接确认)
  0x2C = PSH      (数据传输)
  0x2D = FIN      (断开连接)
```

#### 9.2.3 关键 Bug 修复

**问题**：`OnConnect()` 中调用了 `ForwardTcpConnection()`（一个阻塞式的读取循环），导致：
1. `ForwardTcpConnection` 内部的 `transmission->Read()` 会读取**所有**数据包（包括其他连接的）
2. `OnConnect` 返回 `false` → `PacketInput` 返回 `false` → `Run()` 循环退出
3. 切换连接目标时，整个 Exchanger 被销毁，所有连接中断

**修复**：
1. `OnConnect()` 只负责建立出站连接，发送 CONNECT_OK，然后启动 `ReceiveTcpConnection()`（异步读取），立即返回 `true`
2. `OnPush()` 负责将客户端数据写入出站套接字，返回 `true`
3. `OnDisconnect()` 负责清理连接，返回 `true`
4. 所有错误路径均返回 `true`，确保 `Run()` 循环不会退出
5. `ForwardTcpConnection()` 保留但不再被调用

## 十、日志系统增强（2026-06-23）

> 本章记录 2026-06-23 为增强 WSS 链路调试能力所做的日志系统修改。

### 10.1 功能概述

#### 10.1.1 背景

原始日志系统仅记录 VPN/DNS/ARP/PORT/CONNECT/MP 等基本事件，WSS 握手调试信息仅输出到 `stderr`，客户端没有独立的日志文件。当出现 WSS 连接异常时（如优选 IP 不通、TLS 握手失败、CDN 干扰等），难以定位问题。

#### 10.1.2 解决方案

1. **扩展 `VirtualEthernetLogger`**：新增 9 种日志方法，覆盖握手、TCP 连接生命周期、模块启动等场景
2. **客户端日志支持**：`VEthernetNetworkSwitcher` 增加 `VirtualEthernetLogger` 支持，通过 `client.log` 配置日志路径
3. **双输出日志**：WSS 握手调试信息同时输出到 `stdout`（默认日志）和 `stderr`（终端）
4. **TCP 连接生命周期日志**：服务端 `VirtualEthernetExchanger` 记录 TCP 连接的建立、断开、数据转发

#### 10.1.3 Debug 与 Release 的区别

- debug 文档保留终端调试输出、链路日志、模块启动日志的完整说明。
- release 文档只保留默认日志体系中的链路日志，不将终端调试输出作为标准交付内容。
- 当需要跨 Debian 机器复现问题时，应优先参照本 debug 文档进行环境准备与日志验证。

### 10.2 新增日志方法

| 方法 | 格式 | 说明 |
|------|------|------|
| `Handshake(guid, transmission, host, sni, path, ok)` | `[timestamp][GUID] HANDSHAKE transmission=... host=... sni=... path=... ok=...` | WSS 握手结果 |

## 十一、日志系统增强（2026-06-23 第二轮）

> 本章记录 2026-06-23 第二轮修改：服务端/客户端握手日志、ModuleStart 日志、Executors::Now 时间修复。

### 11.1 修改内容

#### 11.1.1 服务端握手日志

在 `VirtualEthernetSwitcher::Run()` 中，`HandshakeClient()` 返回后记录握手结果：

| 场景 | 日志内容 |
|------|----------|
| 握手失败 (`session_id == 0`) | `[timestamp][00000000-...] HANDSHAKE FAILED REMOTE:client_ip:port` |
| 握手成功 (`session_id != 0`) | `[timestamp][session_guid] HANDSHAKE OK REMOTE:client_ip:port` |

**修改文件：**
| 文件 | 修改内容 |
|------|----------|
| `ppp/app/server/VirtualEthernetSwitcher.cpp` | `Run()` 中 `HandshakeClient()` 返回后调用 `logger->Handshake()` 记录成功/失败 |

#### 11.1.2 服务端 ModuleStart 日志

在 `VirtualEthernetSwitcher::Open()` 中，`OpenLogger()` 之后记录模块启动：

**修改文件：**
| 文件 | 修改内容 |
|------|----------|
| `ppp/app/server/VirtualEthernetSwitcher.cpp` | `Open()` 中 `OpenLogger()` 后调用 `logger->ModuleStart("VirtualEthernetSwitcher", ...)` |

#### 11.1.3 客户端握手日志

在 `VEthernetExchanger::ConnectTransmission()` 和 `Loopback()` 中，`HandshakeServer()` 返回后记录握手结果：

**修改文件：**
| 文件 | 修改内容 |
|------|----------|
| `ppp/app/client/VEthernetExchanger.cpp` | `ConnectTransmission()` 中 `HandshakeServer()` 后调用 `logger->Handshake()`；`Loopback()` 中 `HandshakeServer()` 后调用 `logger->Handshake()` |

#### 11.1.4 Executors::Now 时间修复

**问题**：`ExecutorsInternal` 构造函数未初始化 `Now` 成员，tick 线程首次更新（~10ms 内）前 `Executors::Now()` 返回默认值 `0001-01-01`，导致 `ModuleStart` 等早期日志时间戳异常。

**修复**：在 `ExecutorsInternal` 构造函数的初始化列表中添加 `Now(DateTime::Now())`。

**修改文件：**
| 文件 | 修改内容 |
|------|----------|
| `ppp/threading/Executors.cpp` | `ExecutorsInternal()` 构造函数初始化列表增加 `Now(DateTime::Now())` |

## 十二、编译产物命名规则

### 12.1 命名格式

```
{原始名称}_{日期}_{时间}
```

| 字段 | 说明 | 示例 |
|------|------|------|
| `原始名称` | 平台/架构标识 | `ppp_linux_amd64` |
| `日期` | 编译日期，格式 `YYYYMMDD` | `20260623` |
| `时间` | 编译时间，格式 `HHMM`（24小时制） | `1505` |

### 12.2 完整示例

```
ppp_linux_amd64_20260623_1505
```

### 12.3 存放位置

编译产物存放在 `builds/{日期号}/` 目录下，例如：

```
builds/6.23/
├── ppp_linux_amd64_20260623_1505    # 2026-06-23 15:05 编译
└── ppp_linux_amd64_20260623_1200    # 2026-06-23 12:00 编译
```

### 12.4 与 Release 目录的关系

- 当前 debug 文档记录的是日常调试产物目录规则。
- 若后续发布流程采用 `builds/releases/{日期号}/`，应以 release 文档为准。
- 若当前构建脚本仍输出到 `builds/6.23/`，则以当前脚本行为为准，不强行改动脚本含义。
